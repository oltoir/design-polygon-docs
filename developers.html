Поднимаем дизайн-полигон под свой проект

Универсальная инструкция: поднять полигон под React/Vue/Svelte-проект, настроить .claude-каркас, встроить полигон в pipeline команды. Сквозной пример — booking-app.

Зачем тебе это

Если ты фронтендер в продуктовой команде, типовой widget забирает 4–8 часов: переписать макет из Figma, прописать токены, отступы, состояния, итеративно править после ревью дизайнера. И так десятки раз.

✅ Что даёт полигон

  • Снимает 70% монотонной верстки
  • Один артефакт, один автор верстки
  • Гарантия токенов и spacing на уровне инфраструктуры
  • Время фронта — на логику и архитектуру

📋 Что требуется от тебя

  • 1–2 дня на первичную настройку
  • Код-ревью PR из полигона (15–30 мин на PR)
  • Перенос верстки в production через Claude
  • Регулярная актуализация .claude/rules

Анатомия дизайн-полигона

Жёсткие границы

✓ Полигон должен

  • Повторять структуру боевого проекта (FSD-папки, naming)
  • Использовать тот же стек (React/Vue + TS + Tailwind)
  • Запускаться локально через npm run dev
  • Использовать те же дизайн-токены, что и production
  • Иметь .claude/ с rules / agents / commands

✕ Полигон не должен

  • Делать fetch, axios, TanStack Query, useFetch
  • Иметь бизнес-стор (Pinia / Zustand / Redux)
  • Подключать реальный роутер с данными
  • Хранить секреты, env-переменные
  • Использовать дефолтные Tailwind-цвета (text-gray-*)
  • Использовать произвольный spacing (p-[13px])
🛑

Это жёсткие правила. Валидируются через .claude/hooks/check-tokens.py и блокируют операцию Claude, если он их нарушил. Это не «guidelines», это инфраструктура.

Связь с production

Полигон — «архитектурный двойник» production:

📦 Production
src/features/search/components/HotelCard/ ├── HotelCard.tsx ├── HotelCard.test.tsx # тест ├── index.ts ├── types.ts └── useHotelCard.ts # API-хук
🧪 Polygon
src/widgets/hotel-card/ ├── HotelCard.tsx # та же разметка │ ├── index.ts # тот же экспорт ├── types.ts # те же типы # нет хука — моки в файле

Сквозной пример: booking-app

Сценарий. Команда строит booking-app — SPA для бронирования отелей. Production-стек: React 18 + TypeScript + Vite + Tailwind v3 + Zustand + React Router + TanStack Query. Нужно поднять полигон booking-design-polygon.

Пример экстраполируется на любой стек. Vue-вариант отличается только синтаксисом компонентов и зависимостями; структура .claude/ и принципы — те же.

2

Архитектура: Feature-Sliced Design

Три слоя, направление зависимостей только вниз:

pages/ роуты верхнего уровня, импортит widgets + shared
▼
widgets/ композитные блоки, импортит только shared
▼
shared/ примитивы (ui, assets, types), не знает про слои выше

Структура папок:

src/ ├── pages/ │ ├── search-results/ │ │ ├── SearchResults.tsx │ │ └── index.ts │ └── hotel-details/ ├── widgets/ │ ├── hotel-card/ │ │ ├── HotelCard.tsx │ │ ├── types.ts │ │ └── index.ts # обязательный экспорт │ └── search-bar/ ├── shared/ │ ├── ui/ # Button, Input, Badge │ ├── assets/ # SVG, изображения │ ├── css/ │ │ └── tokens.css # CSS-переменные │ └── types.ts ├── App.tsx └── main.tsx
⚠️

Правило index.ts: у каждого компонента рядом обязателен index.ts с явным экспортом. Это валидируется hook'ом — если файла нет, операция Claude блокируется.

4

.claude/ — мозг полигона

Без правильного .claude/-каркаса полигон не работает: Claude не знает ваших правил и пишет «как привык» — с дефолтными Tailwind-цветами, без index.ts.

Файл .claude/hooks/check-tokens.py

Это сердце валидации. PostToolUse-hook: после каждого Edit/Write регексами ищет запрещённые конструкции → exit code 2 блокирует операцию.

python.claude/hooks/check-tokens.py
#!/usr/bin/env python3
import json, os, re, sys

FORBIDDEN_COLORS = [
    r'\btext-gray-\d+\b', r'\bbg-gray-\d+\b',
    r'\bbg-white\b', r'\bbg-black\b',
    r'\btext-white\b', r'\btext-black\b',
    r'\b(text|bg|border)-(red|blue|green|yellow|purple|pink)-\d+\b',
]
ARBITRARY_SPACING = re.compile(r'\b[pm][trblxy]?-\[\d+(?:\.\d+)?px\]')

def main():
    payload = json.load(sys.stdin)
    if payload.get('tool_name') not in ('Edit', 'Write', 'MultiEdit'):
        return 0
    file_path = payload.get('tool_input', {}).get('file_path', '')
    if not file_path or not file_path.endswith(('.tsx', '.ts', '.vue')):
        return 0

    with open(file_path) as f: content = f.read()
    if 'claude-allow:tokens' in content.splitlines()[0:1]: return 0

    violations = []
    for pat in FORBIDDEN_COLORS:
        for m in re.finditer(pat, content):
            line = content[:m.start()].count('\n') + 1
            violations.append(f'  L{line}: запрещённый цвет «{m.group()}»')
    for m in ARBITRARY_SPACING.finditer(content):
        line = content[:m.start()].count('\n') + 1
        violations.append(f'  L{line}: произвольный spacing «{m.group()}»')

    # + проверка index.ts рядом с компонентом
    # …

    if violations:
        print('❌ check-tokens.py: нарушения правил полигона:', file=sys.stderr)
        for v in violations: print(v, file=sys.stderr)
        sys.exit(2)

if __name__ == '__main__': main()

Регистрация в .claude/settings.json:

json.claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          { "type": "command", "command": ".claude/hooks/check-tokens.py" }
        ]
      }
    ]
  }
}

Slash-команда /make-template

Двухфазный workflow: уточнение → реализация. Фаза 1 жёстко блокирует написание кода, пока дизайнер не подтвердил summary.

md.claude/commands/make-template.md
---
description: Создать новый компонент с нуля в полигоне.
---

# /make-template

Используй агента design-creator.

## Фаза 1 — Уточнение (ОБЯЗАТЕЛЬНО)
Задай дизайнеру по очереди:
1. Название и назначение
2. Где живёт (FSD-слой и путь)
3. Данные (поля, типы)
4. Состояния (hover, focus, disabled, loading)
5. Адаптивность (mobile / tablet / desktop)
6. Картинка макета — есть ли? Если нет — попроси прикрепить

После всех ответов выдай summary и спроси: «Готов начать? [да / правки]».

ЖЁСТКО: НЕ пиши код, пока дизайнер не подтвердил summary.

## Фаза 2 — Реализация
1. Git: создай ветку feature/<slug>
2. Assets: запроси SVG/изображения если нужно
3. FSD-слой: определи и создай папку
4. Naming: PascalCase-файл, kebab-case-папка
5. Файлы: Component.tsx + types.ts + index.ts
6. Стили: только токены, только 4px-grid
7. Build: npm run build — должно быть зелёным
8. Components-map: добавь строку в docs/components-map.md

Аналогично организованы остальные команды и агенты — смотри референсы внизу страницы.

Pipeline передачи верстки полигон → production

Этап A: дизайнер делает PR

Дизайнер собирает компонент через лаунчер, коммитит в feature-ветку, открывает PR. Подробности — в гайде для дизайнера.

Этап B: код-ревью PR в полигоне

📋 Чек-лист код-ревью

  • FSD-зависимости не нарушены (нет импорта widgets/ → shared/)
  • Только дизайн-токены (нет text-gray-*, bg-white)
  • Spacing кратен 4px (нет p-[13px])
  • У компонента есть рядом index.ts
  • Нет API-вызовов: fetch(, axios., useQuery(
  • Нет store/router-вызовов: useStore, useRouter, navigate(
  • Моковые данные локализованные (нужный контекст команды)
  • Компонент в components-map.md со статусом
  • Скриншот превью приложен к PR (Mobile + Desktop)
  • npm run lint и npm run build зелёные
  • Семантика: <button>, не <div onClick>
  • Все интерактивные состояния отрабатывают (hover, focus-visible, disabled)
  • Нет console.log, debug-кода

Этап C: мерж в main полигона

После прохождения чек-листа — squash-merge. Можно включить branch protection в GitHub: требовать минимум одно approve и зелёные чеки.

Этап D: перенос в production через Claude

Открываешь production-репо в Claude Code и даёшь промпт вида:

promptпример
Перенеси компонент HotelCard из
~/Desktop/booking-design-polygon/src/widgets/hotel-card/
в наш production-репо
~/Desktop/booking-app/src/features/search/components/HotelCard/.

Конвенции production:
- Структура: feature-based (features/search/components/HotelCard/)
- Стор: Zustand, useSearchStore. Карточка получает hotel из родителя через пропс
- Данные приходят через TanStack Query useHotelsQuery в родительской HotelsList
- Токены: те же, что в полигоне (синхронизируем через packages/tokens)
- i18n: react-i18next, ключи в src/locales/ru.json под namespace search.hotelCard
- Тесты: vitest + @testing-library/react, smoke-тест на рендер

Что нужно:
1. Создать HotelCard.tsx, перенести разметку 1:1
2. Заменить моки на типизированный пропс hotel: Hotel из shared/types
3. Текст → t() вызовы, ключи в locales
4. Создать index.ts с экспортом
5. Создать HotelCard.test.tsx

НЕ нужно:
- НЕ добавлять API-вызовы внутрь карточки
- НЕ менять токены и spacing
- НЕ заводить store внутри компонента

📋 Чек-лист переноса в production

  • Структура папок — production, не полигонная
  • API-вызовы добавлены (или вынесены в родителя)
  • Моки удалены
  • Store / router / контекст подключены
  • i18n-ключи заведены
  • Тесты добавлены (минимум smoke-render)
  • Сборка production зелёная
  • Запущено локально на dev-данных, проверено визуально
  • Линтер production зелёный
  • Аналитика подключена (если есть)

Этап E: PR в production-репо

Стандартный процесс команды → ревью → мерж → деплой.

Антипаттерны и грабли

✕ НЕ делать

  • Авто-подтверждать все tool-use Claude (давай Allow once для bash)
  • Позволять полигону отращивать бизнес-логику — это убьёт песочницу
  • Делать огромные CLAUDE.md (>80 строк) — Claude хуже их использует
  • Переносить полигонную верстку 1:1 без адаптации к production-структуре
  • Игнорировать ревью — путь к энтропии
  • Использовать одну сессию чата для разных компонентов

✓ Обязательно делать

  • Синхронизировать токены полигона и production (один источник правды)
  • Раз в спринт чистить полигон от deprecated-компонентов
  • Тренировать дизайнеров первые 1–2 недели — это инвестиция
  • Документировать ADR — защита от «а давайте теперь иначе» через год
  • Делать ревью каждого PR, особенно первый месяц
  • Каждый компонент в своей feature-ветке + свежий чат
